# Бизнес требования - Разработка плагинов WordPress

## Технические требования

### Архитектура системы
- **Платформа**: WordPress
- **Тип разработки**: Плагины для WordPress
- **Метод развертывания**: SSH-соединение с хостингом

### Техническая инфраструктура

#### Соединение с хостингом
- **Протокол**: SSH (Secure Shell)
- **Метод обновления файлов**: Через SSH-соединение
- **Безопасность**: Защищенное соединение для передачи файлов

#### Процесс разработки и развертывания
1. **Создание визуального прототипа** интерфейса плагина
2. **Согласование прототипа** с заказчиком
3. **Разработка плагинов** локально в папке `plagin dev` (только после согласования)
4. **Тестирование** в локальной среде WordPress
5. **Загрузка на хостинг** через SSH из папки `plagin dev`
6. **Активация плагинов** в админ-панели WordPress

#### Структура проекта
- **Папка разработки**: `plagin dev` - основная директория для хранения файлов плагинов в процессе разработки
- **Организация файлов**: Каждый плагин должен иметь отдельную подпапку в `plagin dev`

#### Технические ограничения
- Все файлы должны быть совместимы с WordPress API
- Плагины должны следовать стандартам WordPress Codex
- SSH-соединение требует правильной настройки ключей доступа
- Необходимо обеспечить совместимость с версией WordPress на хостинге

### Требования к разработке
- **Язык программирования**: PHP (основной для WordPress)
- **Дополнительные технологии**: JavaScript, CSS, HTML (при необходимости)
- **Серверные технологии**: Node.js, Python (доступны на хостинге)
- **Структура плагинов**: Соответствие стандартам WordPress
- **Документация**: Встроенная документация в коде плагинов

### Технологический стек хостинга
- **WordPress**: Основная CMS платформа
- **PHP**: Серверный язык для WordPress плагинов
- **Node.js**: Доступен на хостинге для дополнительных серверных задач
- **Python**: Доступен на хостинге для скриптов, автоматизации и обработки данных
- **SSH**: Для управления файлами и развертывания

### Правила разработки
- **Обязательный этап**: Создание визуального прототипа интерфейса перед началом разработки
- **Согласование**: Функционал добавляется только после согласования прототипа с заказчиком
- **Ускорение разработки**: Прототипирование позволяет избежать переделок и ускорить процесс

### Основные правила работы
1. **Правило №1**: НЕ разрабатывать ничего, не уточнив все детали
2. **Правило №2**: Сначала прототип, а потом разработка
3. **Правило №3**: Используем SSH для обмена файлами

### Дополнительные правила работы с заказчиком
- **Обязательное уточнение**: Перед созданием любого файла или прототипа необходимо уточнить все детали задачи
- **Вопросы перед действием**: Всегда задавать уточняющие вопросы о функционале, требованиях и предпочтениях
- **Согласование каждого шага**: Не создавать файлы без предварительного обсуждения и согласования
- **Итеративный подход**: Работать по принципу "уточнить → согласовать → реализовать"
- **Обратная связь**: Регулярно запрашивать обратную связь на каждом этапе разработки

### Безопасность
- SSH-ключи для безопасного доступа к хостингу
- Валидация всех пользовательских данных
- Защита от SQL-инъекций
- Соблюдение принципов безопасности WordPress
